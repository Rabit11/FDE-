二、设计理念

从 FDE 审批到价值流动

维度传统 FDEFDE管理平台
管理对象审批签字、日志条数任务价值流动速度与可视化
任务分类多阶段审批队列执行中 → 阻塞 → 已归档 / 已终止（四态）
领导职责逐级审批签字审核备案 + 流动干预（终止/二次分配/退回）
执行方式填表上报工作台提交进展，完成自动归档
可视化表格工单列表视图（多维筛选）+ 四列 Kanban（拖拽）
AI 角色无需求拆解、人员推荐、风险预警、站会生成

端到端价值流

📤 需求提交→ 🔍 选择审核人+执行人→ 🚀 自动进入执行中→ ⚡ 执行中→ 📤 每日进展→ 🚧 阻塞（可选）→ 📦 已归档 · ⏹ 已终止
  • 执行人员上报卡点 → 任务自动进入阻塞
  • 领导对阻塞任务二次分配 → 退回执行中
  • 领导对执行中任务终止 → 进入已终止（与正常归档区分）
  • 执行人员标记任务完成 → 进入已归档
  • 领导可将已归档/已终止任务退回执行中

三、角色与权限体系

采用 capabilities（能力标签） + role（角色层级） 双层模型，登录时动态生成侧栏导航，API 层统一鉴权。

3.1 三类角色

层级人员role权限
超级管理员 admin（工号 666666） admin 全部功能；团队管理；删除任务；can_manage_users
领导 曾锐、林宇飞、毛研勋、张弛 manager 任务中心、全局看板、需求与 AI、领导操作 API；无超管系统权限

3.2 能力标签

标识名称说明
reviewer审核人员领导/超管；管理看板、审核备案、领导专属 API
proposer需求提出全员；需求提交页
executor执行人员全员；工作台、进展、完成

3.3 鉴权要点

  • requirePermission：all 仅表示超管 bypass，与 review 等并列时领导可通过 reviewer 能力访问
  • checkReviewerAccess：领导（isLeader）可操作所有任务的审核类动作
  • 任务中心状态变更 PATCH：仅 reviewer/admin；执行人员在今日工作台提交进展
  • DELETE /api/items/:id：仅超级管理员

4.2 状态转换规则

触发动作操作者状态变化
需求提交（含审核人+执行人）提出人→ in_progress，生成 REQ-YYYY-NNNN
提交进展（含卡点）主执行/协助→ blocked，记录 blocker_type / blocked_reason
提交进展（无卡点）主执行/协助保持状态，追加 progress_updates
任务完成主执行/协助→ done，acceptance_status=accepted
领导终止领导/超管→ terminated（仅执行中可终止）
领导二次分配领导/超管blocked → in_progress，更新人员，清除阻塞
领导退回执行中领导/超管done/terminated → in_progress，清除 completed_at
看板拖拽 / PATCH领导/超管在四态间映射 status，同步 acceptance_status

五、任务中心（双视图）

v2.0 将原流动看板、审核备案、归档查看合并为任务中心。顶部提供列表/四列看板切换，共用四态筛选与人员筛选。

5.1 视图切换

视图适用场景功能
📋 列表视图按人/状态批量查看、行内快捷操作表格展示 8 列信息 + 可执行操作列
📊 四列看板直观感受流动、拖拽调整执行中/阻塞/已归档/已终止四列 Kanban 卡片

5.2 筛选栏（两视图共用）

  • 状态按钮：全部 · 执行中 · 阻塞 · 已归档 · 已终止（带数量角标）
  • 审核人员下拉：领导可见，按 reviewer 过滤
  • 执行人员下拉：按主执行人或协助人过滤

5.3 列表视图字段（v3.0）

列名数据字段
编号req_no
任务名称title（完整显示，表格自适应宽度）
状态flowCategory → 执行中/阻塞/已归档/已终止
提出人 / 审核人 / 主执行 / 协助人created_by、reviewer、assignee、assistants
开始 / 更新created_at、updated_at
操作详情（全员）+ 终止/分配/退回（领导按状态）

列表采用紧凑字号、单行不换行、整表一屏宽展示；点击行或「详情」打开任务详情弹窗（含进展时间线）。支持 CSV 导出，文件名格式：FDE_{类型}_{筛选}_{导出人}_{时间戳}_{条数}.csv。

5.4 四列看板

  • 每列显示 WIP 计数与上限，超限显示 WIP 告警
  • 领导可拖拽卡片在列间移动（映射四态 status）
  • 卡片含类型标签、状态药丸、人员信息、进展摘要、领导操作按钮
  • 执行人员：仅看自己相关任务，卡片只读不可拖拽

5.5 数据范围

角色可见任务
领导 / 超管全部有 req_no / reviewer / assignee 的任务
执行人员assignee 或 assistants 包含自己的任务

七、领导专属操作

在任务中心（列表行内/看板卡片）、任务详情弹窗中执行：

操作适用状态API效果
⏹ 终止执行执行中POST /api/items/:id/reviewer-terminate→ terminated，归入已终止

九、需求提交流程

输入来源

  • 手动填写：标题、描述、验收标准、审核人、主执行人、协助人
  • 会议记录：文本/录音/文档上传 → AI 解析拆解
  • 语音上传：DashScope ASR 转写 → 自动创建任务
  • 文档上传：PDF/Word → 文本提取 → AI 拆解

提交时若已指定审核人 + 主执行人，任务直接进入执行中，并自动生成 REQ 编号。AI 拆分创建的任务在 v3.0 起同样自动补全编号、审核人与智能分配执行人（启动时历史数据自愈）。页面内置 AI Copilot 辅助完善需求描述与验收标准。

十、AI 智能能力

🧠 大模型 LLM

默认 DeepSeek（deepseek-chat），可选 deepseek-reasoner 深度推理。

用途：需求拆分、Copilot 对话、站会/风险/回顾增强、会议拆解、文档分析、智能分配推荐。

🎙️ 语音 ASR

阿里云 DashScope（paraformer-v2 等），含敏捷热词校正。

用途：会议录音转写、语音需求、长音频异步转写（需 ASR_PUBLIC_BASE_URL）。
